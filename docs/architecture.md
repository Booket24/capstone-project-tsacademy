# Architecture Document

## Overview

This document describes the architecture of the `capstone-project-tsacademy` repository, which implements a production-style deployment of the TaskApp application using AWS, Terraform, kOps, and Kubernetes.

The system is designed to deploy:

- A private AWS VPC with public and private subnets
- A kOps-managed Kubernetes cluster running across multiple Availability Zones
- A private Amazon RDS PostgreSQL database for application storage
- Route53 DNS records for public application endpoints
- TLS termination via cert-manager on the Kubernetes ingress controller
- Secrets integration using the AWS Secrets Store CSI driver

## Core Components

### AWS Infrastructure (Terraform)

The `terraform/root/` stack provisions the core AWS environment:

- VPC with public and private subnets
- S3 bucket for Terraform remote state storage
- DynamoDB table for Terraform state locking
- Billing alert resources
- IAM roles and policies used by the infrastructure

The repository uses modular Terraform code for better organization and reuse.

### Kubernetes Cluster (kOps)

The Kubernetes control plane and worker nodes are managed through kOps, with the following characteristics:

- Private topology for Kubernetes nodes
- Multi-AZ deployment across three Availability Zones
- NGINX ingress controller installed in the cluster
- cert-manager for automated HTTPS certificates
- AWS Secrets Store CSI driver for secret material injection into pods

Cluster configuration is generated by `scripts/kops-setup.sh` and applied by `scripts/kops-start.sh`.

### Database Layer (AWS RDS)

The application database is provisioned through Terraform in `kops/terraform_rds/`:

- Private PostgreSQL instance inside the application VPC
- Database subnet group spanning private subnets
- Instance class and engine version configured through variables
- Secure network access limited to the application cluster

### DNS and Routing (Route53)

Route53 DNS is configured in `kops/terraform_route53/`:

- Hosted Zone lookup for the chosen domain
- CNAME records for `taskapp.<domain>` and `api.<domain>`
- External ingress load balancer DNS name wired into the records

The Kubernetes ingress manifest in `k8s/routing.yaml` routes user traffic:

- `/` → frontend service
- `/api` → backend service

## Kubernetes Application Topology

The application is deployed using Kubernetes manifests in `k8s/`:

- `frontendDeployment.yaml`
  - React frontend deployment and service
  - 2 replicas
- `backendDeployment.yaml`
  - Flask backend deployment and service
  - 2 replicas
  - Resource requests and limits defined
  - Liveness and readiness probes configured
  - Secret material injected via CSI provider
- `routing.yaml`
  - Ingress resource for HTTP(S) traffic
  - Host-based routing to frontend and API endpoints
  - TLS configuration using cert-manager
- `cluster-issuer.yaml`
  - cert-manager issuer definition
- `secretProviderClass.yaml`
  - AWS Secrets Store CSI provider configuration
- `service-account.yaml`
  - Service account for external secrets access
- `secretStore-perm.yaml`
  - Permission manifest for the secrets provider

## Networking Architecture

The networking design includes:

- A VPC with separate public and private subnets
- Public subnets for load balancers, NAT gateways, and cluster control plane entry
- Private subnets for Kubernetes nodes and the RDS database
- Route53 DNS records for frontend and API endpoints
- The cluster configured with private topology via kOps

## Security Design

The architecture emphasizes cloud-native security principles:

- Least-privilege IAM roles for kOps and AWS operations
- Private RDS database access limited to the cluster network
- TLS termination through cert-manager and NGINX ingress
- Secret retrieval using AWS Secrets Store CSI rather than hardcoded values
- Remote Terraform state with locking to prevent concurrent drift

## Deployment Flow

Deployment follows this general order:

1. Provision the core AWS infrastructure with Terraform.
2. Configure and create the kOps Kubernetes cluster.
3. Provision the private RDS database.
4. Install cluster services and deploy the application manifests.
5. Configure DNS routing and TLS for public access.

## Known Architecture Constraints

The current repository implements the core AWS and Kubernetes architecture, but the following areas require additional work for a fully production-ready solution:

- Pod security admission is not explicitly enforced by manifest-level policies.

## Notes

This architecture document reflects the current repository state and intended design. It is intended to guide deployment, extension, and future hardening of the TaskApp production deployment.
